¿Qué es ContaIA y para quién va?

Problema que resuelve · Público objetivo · Contexto argentino

El problema real

El 70% de los monotributistas en Argentina pierde vencimientos fiscales

En Argentina hay más de 4 millones de monotributistas. La mayoría son emprendedores, freelancers y pequeños comerciantes que deben gestionar solos sus obligaciones con ARCA: pagos mensuales, recategorizaciones semestrales, declaraciones juradas y cambios regulatorios constantes. El resultado: deudas evitables, multas e intereses por falta de información o simple olvido.

¿Para quién va la app?

💰 Monotributistas (Categorías A–K)

El usuario principal. Desde el emprendedor que factura $500k al mes hasta el comercio con $40M anuales. Necesita saber cuándo pagar, si debe recategorizarse y qué novedades de ARCA lo afectan.

🏢 Pequeñas empresas y Responsables Inscriptos

Negocios con empleados o mayor facturación que deben gestionar IVA mensual, Ganancias anuales y aportes de seguridad social.

👩‍💻 Profesionales independientes

Diseñadores, programadores, médicos, abogados, arquitectos. Trabajan solos, facturan servicios y muchas veces no tienen asesor contable propio.

🚀 Nuevos emprendedores

Personas que recién se registran en ARCA y no conocen el sistema. ContaIA actúa como guía desde el onboarding hasta la operación diaria.

¿Qué automatiza ContaIA?

Procesos automatizados · Motor de alertas · Integración IA · Arquitectura técnica

Automatizaciones implementadas
1

🔔 Motor de alertas automático — cada 60 minutos

El sistema verifica cada hora todos los vencimientos del calendario ARCA contra la fecha actual. Si detecta algo vencido o próximo a vencer (≤7 días configurables), dispara automáticamente: un banner de alerta en la pantalla, notificaciones toast y un badge en el menú. El usuario no necesita hacer nada.

2

📅 Cálculo automático de estados de vencimientos

Cada vencimiento del calendario ARCA tiene su estado calculado automáticamente en tiempo real: Vencido (fecha pasada + impago), Próximo (≤7 días), Programado (futuro), Pagado (marcado por el usuario). El color del calendario se actualiza solo.

3

🤖 Generación automática de sugerencias personalizadas por IA

Al ingresar al módulo IA, el sistema carga automáticamente el perfil del usuario (tipo de contribuyente, categoría de monotributo, estado fiscal) y genera sugerencias personalizadas sin que el usuario tenga que preguntar nada. Usa el notebook de NotebookLM con toda la info de ARCA cargada.

4

📰 Actualización y clasificación automática del feed de ARCA

Las novedades de ARCA se cargan, clasifican por categoría (Monotributo, IVA, Ganancias, etc.) y se marcan automáticamente como leídas/no leídas. El badge de notificaciones en el menú se actualiza solo mostrando cuántas novedades no leyó el usuario.

5

🔄 Detección automática de necesidad de recategorización

En el Dashboard, el sistema evalúa el estado fiscal del usuario y muestra automáticamente un consejo si detecta que está en período de recategorización (enero/julio) o si declaró que sus ingresos se acercan al límite de su categoría actual.

6

💾 Persistencia automática de sesión y datos

El sistema guarda automáticamente en localStorage: sesión del usuario, empresa, vencimientos pagados, novedades leídas, historial de consultas IA y tareas pendientes. Al reabrir la app, todo está restaurado sin ninguna acción del usuario.

Preguntas que puede hacer el profesor — con respuestas

❓ "¿Por qué usaron localStorage y no una base de datos real?"

Para el alcance del TP, localStorage cumple perfectamente: persiste datos, soporta múltiples usuarios en la misma máquina y no requiere backend. La arquitectura está diseñada para que la capa de storage se pueda reemplazar por una API REST sin cambiar el resto del sistema. En producción usaríamos PostgreSQL o Firebase.

❓ "¿Qué tan real es la integración con NotebookLM?"

100% real. Creamos un notebook en notebooklm.google.com con 5 fuentes de información fiscal argentina (Monotributo 2025-2026, ARCA institucional, Ganancias, Bienes Personales, portal oficial de ARCA). El sistema hace queries reales y devuelve respuestas con citas a las fuentes originales. Se puede verificar en: notebooklm.google.com/notebook/f61ac173-b64f-4b14-b87b-343638438875

❓ "¿Cómo escalaría este sistema a producción?"

1) Backend Node.js/Python con autenticación JWT. 2) Base de datos PostgreSQL para usuarios y datos fiscales. 3) API real de ARCA (cuando esté disponible) o scraping del portal. 4) Sistema de notificaciones push reales (Web Push API). 5) NotebookLM Enterprise API para queries en producción. La arquitectura modular actual lo permite sin reescribir el frontend.

❓ "¿Qué principios de Ingeniería de Software aplicaron?"

Separación de responsabilidades: cada módulo JS tiene una única función (auth, storage, calendar, etc.). DRY: utilidades centralizadas en utils.js y storage.js. Modularidad: cualquier módulo puede reemplazarse sin afectar a los demás. UX/Usabilidad: onboarding guiado, feedback visual inmediato, diseño intuitivo.

✅ Punto fuerte para mencionar: el valor diferencial del TP

A diferencia de sistemas típicos que solo muestran información, ContaIA actúa proactivamente: el sistema trabaja aunque el usuario no esté mirando la pantalla. El checker de alertas corre en background, los estados se recalculan solos y la IA genera recomendaciones sin que nadie pregunte. Eso es automatización real.
